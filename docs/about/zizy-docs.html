Review Zizy Docs

This pages provides you with an overview of Zizy's API documentation.

Welcome!​

At a first glance, the landing page contains too much information. It contains information that could be included in an "About the product" section, such as "How Zizy works". Also, the "What products do Zizy offer?" section looks a bit "lost" in the middle of the landing page, and the offered products aren't highlighted either.

The landing page should be concise and straight to the point, i.e. include a quick sentence explaining what Zizy is and what it does, with cards linking to the relevant products (for a visually appealing experience).

Personally, I don't like the emoji that comes before Welcome!.

Get Started​

This section does not contain enough information for a user to be able to authenticate themselves using API keys, such as JWKs. Instead of putting the burden on the Zizy team, the documentation should clearly explain how JWKs work (assymmetric keys, maybe?) and how to authenticate.

Also, it explains how to make the first request, but it doesn't explain how to make the following ones.

Overall, it lacks a lot of information, which puts a lot of burden on the Zizy team's engineers. This isn't an efficient approach, because instead of documenting the necessary steps for the client to be able to make requests on their own, the Zizy team has to provide support about the same process over and over again.

POST "/transaction"​

The dynamically-generated page includes a swagger integration, which is the best known option to document API. However, the responses look repeated, both in swagger and in the static documentation that follows it. I don't understand in what way the static documentation about responses complements the dynamic one.

Examples​

The examples should be integrated into the dynamically-generated page, such as what Hetzner does. I'm not sure if there were some limitations from Gitbook, but having this information seggregated looks confusing.

FAQ​

Some of the FAQs could simply be part of a "How Zizy Works" page. In addition, some questions are answered with a request for clients to contact the company, instead of explaining in detail what the user needs to understand.

Again, I don't see a need for emojis coming before the title on the sidebar.

Powered by Gitbook​

Even though proprietary tools provide you with a convenient experience (i.e. support from the company that supplied the software), they come at a cost and typically don't allow users to adapt the software according to their needs. On the other hand, open source tools are free. The open source community is typically very collaborative and fosters a creative atmosphere, which encourages users to adapt the software according to their needs.

Search​

The search engine works well and would be a powerful tool for a well documentated site.

Conclusions​

Overall, Zizy API documentation is very small and lacks information that is relevant for the user.

The landing page does not make a good first impression and contains information that should not be there, such as the "How Zizy works" section. The information could be presented in a more highlighted manner, with cards pointing to the offered products. Also, this page hasn't been maintained for over 10 months, which may indicate that Zizy hasn't increased its portfolio, or that documentation isn't one of Zizy's priorities.

Going though the documentation, I didn't find any information on the products announced on the landing page.

The API documentation looks confusing and seggregated in some parts.

The FAQ could be removed if the information was integrated into the documentation.

The support from Zizy teams could be mitigated if the Zizy API documentation was well documented.
